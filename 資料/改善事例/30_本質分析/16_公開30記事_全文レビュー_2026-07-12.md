# 公開30記事 全文レビュー

レビュー日: 2026-07-12

対象: `case-studies/index.html` から公開されている30記事。`index.html` と `_template.html` は採点対象外。

## 結論

- 残す上位15件は、見積、シフト、月次経理、修繕受付、問い合わせ、営業、採用など、中小企業で発生頻度と負担が大きい業務を具体的に扱えている。
- 記事群の最大の弱点は、同じ構成と定型文の反復、近接テーマの記事重複、出典と公開記事の業務変換距離である。
- 直ちに削除する記事はない。統合先へ固有情報を移してから非公開化する候補が5件、追加調査や大幅改稿を行わないなら非公開化を検討する候補が3件ある。
- `insurance-claim-intake.html` は内容が強い一方、出典台帳がない。削除より先に、保険事故受付の一次情報または導入事例を結び付ける価値が高い。

## 採点基準

各項目を0〜5点、合計30点で採点した。

| 項目 | 5点の基準 | 0〜1点の目安 |
|---|---|---|
| 中小企業経営者への有用性 | 頻度、負担、売上・顧客・資金・人員への影響が大きく、自社へ置き換えやすい | 対象が狭い、経営上の効果が見えない |
| 具体性 | 人物、時刻、入力資料、AIの出力、人の承認点、改善後の行動がそろう | 抽象的なAI活用説明にとどまる |
| 実現性 | 一般的なツールと小さな運用変更で試せ、データ連携や安全面も現実的 | 大規模連携や高精度判断を暗黙に要求する |
| 出典距離 | 同種業務の公式な導入実績や公的事例が記事主張を直接支える | 出典なし、制度説明だけ、元事例と記事業務が大きく離れる |
| 重複の少なさ | 他記事と異なる業務判断、入力、成果を持つ | 別記事と入力・出力・場面・出典がほぼ同じ |
| 記事品質 | 見出し、場面、因果、文章が自然で、数値と安全表現にも問題がない | 定型反復、矛盾、誤記、根拠のない成果断定が目立つ |

「出典距離」は出典自体の知名度ではなく、公開記事の主張をどれだけ直接支えているかで採点した。大企業の導入事例を中小企業向けに変換している場合や、製品説明・一般解説だけの場合は減点した。

## 残す上位15

次の15件は現状でも記事群の主力にできる。順位が同点の場合は、出典の近さ、対象業種の広さ、記事群の構成バランスで並べた。

1. `quote-flow.html` - 町工場の社長が毎日抱える見積前処理を、国内事例に近い形で具体化している。
2. `store-shift.html` - 店舗経営者に痛みが伝わりやすく、小さく試す条件も明確。
3. `accounting-reconciliation.html` - 月末の消込という専門作業を、例外の種類まで具体化している。
4. `property-maintenance.html` - 夜間修繕の焦り、質問、業者連絡、安全判断を一つの場面で追える。
5. `sales-follow.html` - 売上へ近く、商談直後から当日フォローまで再現しやすい。
6. `recruiting-flow.html` - 候補者を待たせる問題へ直結し、採否をAIから切り離せている。
7. `customer-support.html` - 問い合わせの量ではなく、強い苦情が埋もれる問題を扱えている。
8. `recruiting-scorecard.html` - 面接官ごとのばらつきを、会議用の根拠へ変える価値がある。
9. `field-report.html` - 建設・点検・設備など対象が広く、写真から報告・連絡へ進む流れが明快。
10. `procurement-compare.html` - 価格以外の条件を横並びにし、承認説明まで短くする実務性が高い。
11. `accounting-check.html` - 支払前照合は資金とミス防止へ直結し、入力資料と承認点が具体的。
12. `delivery-dispatch.html` - 朝の変更対応を安全・顧客連絡・運行指示へ分けている。
13. `hotel-multilingual.html` - 多言語返信を申し送りへつなぎ、翻訳記事だけで終わっていない。
14. `manufacturing-quality.html` - 不良写真を翌朝の原因・対策会議へつなぐ場面が現場的。
15. `care-handoff.html` - 高リスク領域でも、AIの準備と資格者の判断を具体的に分離している。

## 統合候補

### 1. 問い合わせ対応 + EC返品

- 親記事: `customer-support.html`
- 統合元: `ec-returns.html`
- 移す内容: 返品期限、写真不足、注文番号照合、返金・交換の承認点。
- 根拠: 両記事とも、問い合わせ本文と注文履歴を読み、返信案と責任者案件を分ける。同じ出典群も使っている。
- 統合後: `ec-returns.html` は非公開候補。

- 根拠: 手帳、音声メモ、メール、CRMを入力し、メール案と次回接点を出す流れが共通。出典も重なる。
- 統合後: `sales-lost-follow.html` は非公開候補。

### 3. クリニック受付 + 診察後フォロー

- 親記事: `clinic-reception.html`
- 統合元: `clinic-followup.html`

### 4. AIマニュアル + 店頭在庫FAQ

- 親記事: `shop-manual.html`

- 注意: 在庫数を答えるには最新在庫との連携が必要。静的なFAQとして統合せず、更新日時と参照元を必須にする。
- 統合後: `retail-inventory-faq.html` は非公開候補。

### 5. 店舗シフト + ボランティア配置

- 移す内容: 初参加者への配慮、避けたい作業、持ち物連絡、当日欠席時の代替。
- 根拠: 希望時間、経験、必要人数から配置候補を作り、人が公平性と個別事情を直す構造が共通。
- 統合後: `npo-volunteer.html` は、NPOを主要顧客層として残す方針がなければ非公開候補。

## 横断的な記事品質の問題

1. 30記事中25記事で「作業時間だけでなく、抱えていた不安も軽くなります。」が完全一致している。個別の不安や改善後の行動へ書き分ける必要がある。
2. 多くの記事が「場面・変更・実感」「Before・After・Result」「整理後3枚」「導入前3項目」の同一構成で、全文を読むとテンプレート感が強い。
3. `clinic-reception.html` と `clinic-followup.html` には「判断から始める」という抽象的で不自然な利益表現が残る。
4. `shop-manual.html` の「人とAIの分担」に、`store-shift.html` と同じシフト記事の見出しが入っている。
5. `bcp-checklist.html` には、避難方法と休業範囲を責任者が決める説明が連続して重複する。
6. `hotel-multilingual.html` では、未返信へ早く返せるという同じ文が近接して重複する。
7. 機械チェックは全31 HTMLで通過したが、上記の意味重複、誤記、出典距離は検出されない。

## 推奨する実施順

1. 上位15件を一覧上で優先表示し、まず主力記事群を確定する。
2. 明白な誤記と根拠不足を修正する。対象は `shop-manual.html`、`field-report.html`、`insurance-claim-intake.html`。
3. 5組を統合し、統合元の固有情報が親記事へ移ったことを文単位で照合する。
4. 統合元5件を一覧から外す。HTML削除はアクセス実績とリダイレクト方針を決めてから行う。
5. BCP、農業、飲食の3件は追加調査の期限を決め、直接事例が得られなければ非公開化する。
6. 残る記事の定型文を記事ごとに書き分け、同一構成の反復を減らす。
