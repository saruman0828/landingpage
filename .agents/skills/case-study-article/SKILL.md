# 事例記事作成

## 目的

`case-studies/` の記事を、AIツール紹介ではなく「読者が自社の現場を思い浮かべ、最初の1業務を相談したくなる匿名改善ケース」にする。元事例のコピーではなく、公開情報や既存資産から再利用できる業務改善パターンを抽出し、日本の中小企業向けに具体化する。

## 正本と編集単位

- `case-studies/*.html` を公開記事の正本として扱う。
- JSON、Python辞書、CSVなどの固定データ構造を正本にしない。
- `title`, `lead`, `meta`, `image`, `caption`, `narrative`, `before`, `flow`, `ai`, `human` のような項目名へ記事を無理に固定しない。必要な情報は、既存HTMLのセクション構造、見出し、本文、画像、CTAの中で自然に表現する。
- 新規記事は、近い既存記事のHTMLをコピーして作り、本文・画像・セクション構成を記事ごとに調整する。
- 新規記事の雛形作成には `npm run new:case-study -- <slug>` を使う。近い既存記事から始める場合は `npm run new:case-study -- <slug> --from <existing-case.html>` を使う。
- 大量生成スクリプトや一括再生成スクリプトは使わない。共通化よりも、公開HTMLを読んだ時の自然さ、具体性、安全表現を優先する。
- 補助スクリプトは、HTMLファイルを作るだけにとどめる。記事本文や一覧カードの中身は自動生成しない。
- 既存HTMLを直接編集する場合は、変更した本文・リンク・画像の差分を確認し、意図しない周辺変更を残さない。

記事の価値は、次の順で作る。

1. 読者が毎週困っている仕事を言語化する。
2. AIへ渡す材料、AIが整える出力、人が承認する判断を分ける。
3. 一人の主人公が楽になる場面を描く。
4. 事業面の見返りと感情面の見返りを両方示す。
5. 業種に合う画像で「これは自社にもありそうだ」と感じさせる。

読者の反応を説明語として見出しや本文に直書きしない。共感は「よくある場面」を描いて作り、納得は「何をどう変えたか」を順番に見せて作る。

## 作業前に必ず読むもの

- 記事ページを書く、または修正する前に `references/article-pattern.md` を読む。
- 画像を選ぶ、差し替える、生成指示を作る前に `references/image-briefs.md` を読む。
- 複数記事を追加・整理する場合は、`資料/改善事例/`、`case-studies/`、`case-studies/construction-ai-case-assets/`、必要に応じて画像候補フォルダを確認する。

## 本質設計シート

HTMLを書く前に、各記事ごとに次を短く決める。これが薄い場合は、先に調査か分解をやり直す。

- 読者: 誰が詰まっているか。例: 社長、店長、現場責任者、後継者、受付責任者、営業担当。
- 痛む時間帯: いつ重くなるか。例: 朝一番、現場から戻った夕方、月末、商談直後、閉店後。
- 止まる仕事: 業種名ではなく作業名で書く。例: 写真整理、見積前処理、希望休調整、返品返信、申し送り。
- 隠れたストレス: 不安、割り込み、探し物、見落とし、夜の作業、説明の繰り返し。
- AIへ渡す材料: メール、写真、PDF、Excel、チャット、予約、注文、過去メモ。
- AIの出力: 下書き、一覧、差異、例外、チェックリスト、確認質問、返信案。
- 人間の承認点: 金額、契約、安全、医療、法務、個人情報、外部送信、顧客感情、最終判断。
- 結果: 分単位・時間単位・返信速度・割り込み減少・早期完了など、体感できる変化。
- 戻る時間の使い道: 顧客対応、現場判断、スタッフ教育、営業、品質改善、家族との時間。
- 画像の役割: 主人公、職場、手元、Before/After、成果物、安心感のどれを補強するか。

## 段落設計

各記事の本文は、少なくとも次の3段落の役割が分かるように作る。見出しや本文に読者の反応をそのまま書かず、場面と変化で伝える。

1. 現場で起きていたこと: 誰が、いつ、何を抱えていたか。読者が自社の似た場面を思い浮かべられる具体性を入れる。
2. 変えたこと: AIへ渡す材料、AIが作る下書き・一覧・例外、人が承認する点を分ける。
3. 改善後に残るもの: 何が短くなり、何が減り、戻った時間を何に使えるかを書く。

避けること:

- 共感や納得を示す短い感嘆語を、そのまま見出しや本文に置く。
- 「AIが整理します」で止める。
- 「何がどう変わったか」がないままナラティブだけを書く。
- すべての記事で同じ段落、同じ締め方にする。

## CTA設計

記事末尾や一覧ページのCTAは、読者に「申し込ませる」文ではなく、最初の1業務を軽く見直す入口として書く。

良い型:

- Eyebrow: `相談`
- H2: `自社業務を自動化する` 
- Button: `30分無料診断`

避けるCTA:

- `次は自社の業務で考える`
- `御社なら、どの作業をAIへ渡せるか。`
- `まず、社長や担当者に戻っている1作業を一緒に見つけます。`
- `いきなり全社導入ではなく...`
- ボタン文言の `30分無料診断を申し込む`

CTAは短く、押しつけず、具体的な材料から始める。`御社なら` のように問い詰める形より、`毎週止まっている作業`、`ひとつだけ見える化` のように負担を下げる言い方を優先する。

## 担当分離と文単位レビュー

記事本文の品質改善では、main agent が本文を直接まとめ直さない。本文の改善は必ず「1記事 = 1担当」の単位で扱い、担当サブエージェントが文単位で `現文 -> 改善案` を出す。

守ること:

- 1つの記事本文には、原則として1人の担当サブエージェントを置く。
- 担当サブエージェントは、段落ごとの要約や全面リライトではなく、文単位で `現文 -> 改善案` を提示する。
- main agent は、担当サブエージェントの改善案を統合するだけにする。
- main agent は、記事本文を自分で要約、再構成、言い換え、補筆しない。
- 統合中に違和感、重複、文脈ズレ、表現の弱さを見つけた場合も、main agent がその場で直さず、担当サブエージェントへ該当文を戻す。
- 複数記事を扱う場合は、記事ごとに担当を分ける。別記事の担当者が本文を横断的に混ぜて書き換えない。
- 文を追加する場合も、担当サブエージェントが `追加位置 -> 追加文案 -> 追加理由` を明示する。
- 文を削除する場合も、担当サブエージェントが `現文 -> 削除案 -> 削除理由` を明示する。
- main agent が行ってよいのは、HTMLへの反映、リンク・画像パス・クラス名の整合、表記ゆれの機械的確認、担当案同士の衝突確認までとする。

提出形式:

| 記事 | セクション | 文ID | 現文 | 問題点 | 改善案 | 担当 |
|---|---|---:|---|---|---|---|

禁止:

- main agent による本文修正。
- 段落単位、記事単位の勝手な要約。
- 複数担当の案を main agent が混ぜて新しい本文を作ること。
- 原文が残らない形の全面差し替え。

## 標準ワークフロー

1. 既存記事、既存画像、出典メモを確認し、同じ業務パターンや画像の重複を把握する。
2. 読者の困りごとを「業種」ではなく「毎週止まる仕事」に変換する。
3. 必要なら現在公開されている事例を調査する。一次情報、公式事例、行政・公的機関、信頼できる報道を優先する。
4. 新しい出典、数値、主張を使う場合は、再利用できるメモを `資料/改善事例/` に保存する。
5. 本質設計シートを作り、Before pain -> AI receives materials -> AI drafts/organizes -> Human approves -> Result -> Why it feels good の流れを確認する。
6. `npm run new:case-study -- <slug>` で `case-studies/<slug>.html` を作る。近い既存記事がある場合は `--from <existing-case.html>` を使う。
7. `references/article-pattern.md` に沿って、既存の `blog-*` クラスで記事HTMLを書く。セクション数や見出しは記事内容に合わせて調整してよい。`TODO_CASE_STUDY` は公開前に必ず消す。
8. `case-studies/index.html` に、読みたくなるカードを追加・更新する。カードは業種名より「止まっていた仕事」と「改善後の実感」を前に出す。
9. 画像を既存資産から選ぶ。足りない場合は `references/image-briefs.md` に沿って画像ブリーフを追加する。公開HTMLに存在しない画像パスを置かない。画像の `alt`、`figcaption`、画像横カードの説明は、実写の細部を断定せず、業務改善イメージとワークフロー上の役割で書く。
10. 最後にリンク、画像パス、必須セクション、安全表現、スマートフォンでの読みやすさを検証する。

## 複数記事を追加・整理するとき

大量の記事では、薄いテンプレ量産になりやすい。個別にHTMLを書き始める前に、5件ずつの束へ分け、現場系、事務系、顧客対応系、高リスク領域を混ぜる。

1. 既存記事、`資料/改善事例/`、`case-studies/construction-ai-case-assets/`、必要に応じて画像候補フォルダを確認し、既存の事例・画像・出典メモを再利用する。
2. 各記事ごとに本質設計シートを持たせる。固定フィールドへ埋めるのではなく、主人公、痛む時刻、止まる仕事、人間承認点、画像の役割をHTML上で自然に表現する。
3. 公開記事へ入れる前に、調査結果を「業種」ではなく「毎週止まる仕事」に変換する。例: クリニックではなく問診とフォロー、宿泊ではなく多言語問い合わせと引き継ぎ。
4. 追加記事の対応表は `資料/改善事例/NN_追加XX件_匿名ケース対応表.md` に残す。公開HTMLには出典URLを入れない。
5. 同じ構造の記事でも、記事データからHTMLを一括生成しない。既存HTMLをテンプレートにして個別に作り、後半記事ほどテンプレート臭、画像の弱さ、読後の行動不足を読む。
6. 画像がまだない場合、公開HTMLでは既存の実在画像を使い、画像生成指示だけを別に残す。リンク切れを公開ページに残さない。
7. 最後に、全HTMLの内部リンク、画像パス、必須セクションを機械的に検証する。

## 品質基準

「AIで効率化できます」と言うだけの記事は公開しない。次を満たす。

- 時間帯、場所、人物、作業、困りごとが見える日常場面を入れる。
- `現場の一場面`、`朝の受付で起きていたこと`、`商談後の18時` のような、読者に見せる場面セクションを1つ入れる。内部設計ではナラティブと呼んでよいが、公開見出しに「ナラティブ」と出さない。
- 読者が思い浮かべられる主人公の役割を置く。名前付きの人物や企業は創作しない。
- 分単位、時間単位、返答速度、質問減少、フォロー漏れ減少、早期完了など、体感できる改善結果を入れる。
- 感情面の見返りを入れる。何が楽になるか、何が安心に変わるか、何に時間を戻せるかを示す。
- 事業面の見返りを入れる。返答の速さ、ミスの減少、割り込み減少、見積もりの早期化、教育改善、顧客対応時間の増加などを示す。
- 人間が承認するポイントを具体的に入れる。
- AI導入による改善。

ナラティブは次を含む型を使っても良いが、自然文て記述する。

`Time and place,protagonist under pressure,small concrete friction,AI handoff moment,prepared output appears,human adds judgment,changed feeling`



## ダサくなる兆候

次が出たら、公開前に直す。

- H1が「AI活用事例」「業務効率化」だけで、具体的な変化がない。
- 文章が「AIが整理します」「AIが下書きします」の繰り返しで、人物の場面がない。
- 共感や納得を表す言葉をそのまま置き、肝心の場面や変化が書かれていない。
- 「何を渡す」「何が出る」「人は何を見る」が1段落で分からない。
- 業種名だけが違い、本文の構造と表現がほぼ同じ。
- 画像がロボット、抽象ネットワーク、汎用オフィスで、記事の業務場面を補強していない。
- 数値が大きすぎる、または根拠のない成果断定になっている。
- 高リスク領域で、AIが最終判断、外部送信、法的・医療・安全判断をしている。
- 読後に「何から始めればよいか」が見えない。
- CTAが営業色の強い申し込み文になっている。
- 公開見出しに `ナラティブ`、`読みどころ`、`自社に置き換える` など制作側の言葉が出ている。

## 頻出禁止の言い回し

次の表現は、1記事内や複数記事で繰り返すとテンプレート臭、直訳感、説明臭さが出る。完全禁止ではないが、見つけたら具体語へ置き換える。特に見出し、CTA、記事末の締めで連発しない。

直近の公開HTML集計では、次の順に多かった。対象は `index.html`、`training.html`、`thanks.html`、`case-studies/*.html`。`case-studies/_template.html` は除外。

- `確認` の多用。文脈に応じて `見る`、`承認する`、`照合する`、`見直す`、`目を通す`、`送信前に整える`、`金額を見る`、`契約条件を決める` などへ分ける。
- `人が確認する判断`、`人が確認`、`人が判断`。安全表現として必要な場合でも、`金額だけ社長が見る`、`送る前に担当者が目を通す`、`契約条件は人が決める` のように、誰が何を見るかで書く。
- `材料を整える`、`材料をそろえる`、`材料`。抽象語で止めず、`写真`、`メール`、`見積メモ`、`注文内容`、`問い合わせ履歴`、`PDF`、`Excel` など実物名にする。
- `分けます`、`並べます`、`そろえます`、`整えます` の連発。`一覧にする`、`差分を出す`、`優先順にする`、`送信前の文面にする`、`チェック項目にする` など成果物で書く。
- `先に並べる`、`先に出す`、`先に`。何を先に出すかを明記し、`朝の時点で未返信だけ出す`、`見積に必要な写真を先頭に集める` のように場面化する。
- `探す仕事から`、`探し物`、`探す`。便利な比喩だが多用しない。`注文履歴を開き直す`、`写真フォルダを行き来する`、`過去メモを見返す` など具体動作にする。
- `AIへ渡す`、`AIに渡す`。コンセプト語として使いすぎない。必要に応じて `写真を読み込ませる`、`メール本文を入れる`、`Excelをもとに一覧を作る` など操作に寄せる。
- `同じ場所`、`同じ順番`、`同じ画面`。単調になりやすい。`案件ごとの一覧`、`送信前チェック欄`、`店長用の確認リスト` のように画面や表の役割で書く。
- `見える`、`見える形`、`見えるよう`。抽象的なので、`一覧になる`、`差分が分かる`、`未返信だけ残る`、`赤字で分かる` など、読み手が想像できる結果にする。
- `時間を使える`、`時間を使う`。`戻る` の置き換えとして増えやすい。`朝のうちに電話できる`、`スタッフ教育に回せる`、`次の見積に入れる` のように使い道を書く。
- `形にする`、`形へ`、`形で`。制作側の整理語に見えやすい。`返信案にする`、`チェックリストにする`、`比較表にする` など名詞で受ける。
- `短くなったのは`。記事末の定型にしない。何が減ったかを直接書く。例: `閉店後の入力が減り、翌朝の確認から始められます。`
- `AIの役割は`。見出しの型が見えるため、`AIで下書きまで作る`、`写真から不足だけ拾う`、`問い合わせを返信案にする` のように作業名で書く。
- `ことです`、`ためです`、`ものです`。締めが硬くなる。短く言い切るか、場面で終える。
- `状態にする`、`状態へ`。抽象的なので、`一覧で見られる`、`送信前に読み返せる`、`朝の会議で配れる` のように使う場面を書く。
- `変わりました`、`変える`、`変わる`。結果説明が薄くなりやすい。何が何分短くなったか、誰の割り込みが減ったかを書く。
- `戻る`、`戻す`、`戻せる`。ナビゲーションや物理移動はよいが、時間や確認の比喩では不自然になりやすい。`集まる`、`空く`、`回せる`、`進められる`、`取り戻せる` など文脈で選ぶ。
- `自動化`、`効率化`、`省力化`。汎用語だけで終えない。`請求書チェックを先に終える`、`問い合わせ返信を閉店前に返す` のように具体化する。
- `最後に判断`、`最後に確認`。AI文脈の定型に見える。高リスク領域では `契約条件は担当者が決める`、`外部送信前に責任者が読む` のように承認点を明確にする。

## 調査基準

- 公式の顧客事例、行政や公的機関の事例、ベンダー事例、学術論文、信頼できる報道を優先する。
- ベンダーが示す数値は、確定した成果ではなく主張として扱う。
- 記事内で出典を明示しない場合、具体的な出典数値はそのまま使わず、妥当な規模感へ抽象化する。
- 大企業向けの主張は、中小企業に役立つ単位へ変換する。例: 1人あたり、1週間あたり、1か月あたり、繰り返し作業1回あたり。
- 高リスク領域では保守的に扱う。法務、医療、金融、安全、契約、価格、外部送信では、AIは準備だけを行い、最終判断は人間が持つ。

## 匿名化ルール

- 実名企業は、`ある店舗`、`ある町工場`、`ある営業組織`、`ある現場型企業` のように置き換える。
- 製品名は一般名に置き換える。例: AIチャット、AI作業環境、社内チャット、予約システム、発注画面、OCR、RPA。
- ユーザーから求められない限り、公開HTML内に出典URLを入れない。出典URLは調査メモまたは最終報告に残す。
- 元資料の表現を再現しない。このサイトの文体に合わせて言い換える。

## 文体

- 技術に詳しくない中小企業の経営者向けに日本語で書く。
- 役に立ち、具体的で、少しうらやましく感じられる記事にする。
- 「自分でもAIで自動化できた」を強調する。
- 「AIに聞く」よりも「AIに仕事を渡す」を強調する。
- AIが役立つ場面を、読者が自分で再現できるイメージが湧くように表現する。
- 戻ってきた時間を何に使えるかを示す。
- 誇張を避ける。魔法のように見せない。完全自動化や人間の代替という構図にしない。
- 抽象的な説明より、鮮明な業務場面を優先する。
- 読者に「これ、うちもある」と感じる瞬間を作り、その後に「この状態なら欲しい」と思える瞬間を作る。
- 場面セクションは主張ではなく場面で書く。机の上の物、時刻、割り込み、AIへの具体的な渡し方を入れる。
- 読み手にノウハウ、役立つ情報を提供する。
- 見出しや本文に制作側の言葉や説明語を出さない。

## 検証

最低限、次を確認する。
- 読み返して、不自然な表現、誤字脱字を修正する。
- `npm run check:case-studies` を実行し、画像周辺表現、画像パス、HTMLリンク、CTA禁止語を機械的に確認する。
- `TODO_CASE_STUDY` が残っていない。
- 新規記事が `case-studies/index.html` からリンクされている。
- `href="*.html"` が存在するページを指している。
- `src="..."` が実在する画像を指している。
- 新規・更新記事に場面セクション、`人とAIの分担`、自社で始める手順がある。
- 高リスク領域でAIが最終判断する表現になっていない。
- 一覧ページ `case-studies/index.html` から記事へリンクしている。
- 代表記事だけでなく、後半の記事も読み、テンプレート臭、画像の弱さ、読後の行動不足がないか確認する。
- 公開HTMLと差し込みJSに `ある日のナラティブ`、`読みどころ`、`御社なら、どの作業をAIへ渡せるか。`、`30分無料診断を申し込む` が残っていない。
