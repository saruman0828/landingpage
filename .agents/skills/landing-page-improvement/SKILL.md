# ランディングページ改善

## 目的

訪問者が一度で「誰向けか、何が変わるか、次に何をすればよいか」を理解でき、改善結果を数値で比べられる状態を作る。見た目だけ、文章だけ、計測だけで終えず、問い合わせ完了まで通して扱う。

## 作業前

1. `git status --short`で既存変更を把握し、無関係な変更を上書きしない。
2. `index.html`、`training.html`、`thanks.html`、`style.css`、`assets/contact-form.js`、`assets/analytics.js`、`assets/site-layout.js`、`api/analytics*.js`、`vercel.json`を必要な範囲で読む。
3. 現在の訴求、主要CTA、フォーム、事例導線、計測イベント、公開先を一覧にする。
4. ユーザーが不要としたSEO設定や外部サービスを追加しない。

## 分かりやすい文章

- 内部作業ではなく、読者の場面と利益を書く。
- `整理する`、`仕組みにする`、`判断基準`、`効率化`だけで終えない。誰が何を見て、どの仕事を自分で進められるかを書く。
- 専門語は一般語へ直すか、その場で説明する。例: `指示設計`は`AIへの頼み方`、`原型`は`試作品`。
- 主語と目的語を置く。`人が判断する`ではなく、`店長が返金を承認する`と書く。
- 申し込み前に追加作業を要求しない。日程候補などは、申し込み後に運営側から案内する。
- 根拠のない金額換算、削減時間、成功断定を置かない。数値には算出条件か注記を付ける。
- 見出し、リード、CTA、フォーム周辺を続けて読み、同じ説明の反復を減らす。

文章変更後は`body[data-copy-version]`の版を上げる。例: `main-clear-v1`から`main-clear-v2`。一度に見出し、説明、CTAを全部変えず、比較したい要素を1つに絞る。

## 計測

共通実装は`assets/analytics.js`へ置き、ページごとの重複実装を増やさない。個人名、メール、電話番号、相談本文などの個人情報をイベントへ含めない。

最低限、次を扱う。

- `page_view`: ページ表示
- `case_view` / `case_open`: 事例閲覧と事例導線
- `cta_click`: CTAクリック
- `form_start`: 入力開始
- `form_submit_success` / `form_submit_error`: 送信結果
- `lead_completed`: サンクスページ到達
- `scroll_depth`: 50%・90%読了
- `copy_view`: 文言版

Vercel Web Analyticsは`/_vercel/insights/script.js`を使う。`window.va("event", { name, data })`形式で送り、カスタムプロパティは基本2個以内に抑える。問い合わせ完了はボタン押下ではなく`thanks.html`到達で数える。

新規ページは`assets/site-layout.js`または`assets/analytics.js`を読み込ませる。既存の`/api/analytics`送信は補助ログとして残してよいが、一時ファイルを長期保存先とみなさない。

## 数字から改善する

最初に次の流れを見る。

`page_view -> case_view -> cta_click -> form_start -> lead_completed`

- 冒頭離脱が大きい: H1とリードを短く具体化する。
- 事例閲覧が少ない: 読者に近い事例をCTA前へ出す。
- CTAクリックが少ない: 相談で得られるもの、料金感、安心材料を補う。
- 入力開始後の完了が少ない: 項目数、必須条件、エラーを見直す。
- 90%読了が低い: 重複説明を削り、重要内容を前へ移す。

原則14〜28日、または比較に足る訪問数が集まってから版別に比べる。アクセスが少ない時期に毎週文言を変えない。運用詳細は`analytics-improvement-guide.md`を参照する。

## 画像

- 新規JPEG/PNGをそのまま公開しない。WebPへ変換する。
- 写真は品質92を基準にし、QR、図表、ロゴ、透明画像は可逆圧縮を使う。
- 縦横サイズ、透明部分、画像の役割を保つ。
- 元画像を削除する前に、HTML、CSS、JS、Markdown内の参照を`.webp`へ更新する。
- 変換後が元画像より大きい場合は、圧縮方法を見直す。
- 全画像を開いて破損を検査し、ローカル参照切れが0件であることを確かめる。
- 表示幅別画像は、実測で読み込みが問題になった場合だけ追加する。

## 検証と本番反映

1. JSへ`node --check`を実行する。
2. `npm run check:case-studies`を実行する。
3. HTMLの内部リンクと画像参照がすべて存在するか検査する。
4. ローカルで主要ページ、事例、計測APIが200を返すか検査する。
5. ユーザーが本番導入を求めた場合は既存Vercelプロジェクトへ反映する。
6. Web Analyticsを有効化し、再デプロイ後に`/_vercel/insights/script.js`が200を返すか検査する。
7. 本番の主要ページとランタイムエラーを検査する。

canonical、構造化データ、GA4、Cookie型追跡、セッション録画は自動追加しない。ユーザーの目的と同意を得てから扱う。

## Google Analytics以外の選択

- Vercel Web Analytics: ページ、参照元、基本イベント。Vercel運用なら最初に選ぶ。
- Umami: ファネル、経路、UTM、セッション分析が必要な場合。
- Plausible: シンプルな集客・目標・ファネルを重視する場合。
- PostHog: セッション再生、実験、高度な行動分析が必要な場合。

複数ツールを最初から入れない。Vercelで不足する分析項目を明確にしてから追加する。

## 完了条件

- 重要な公開文が具体的で、専門語に説明がある。
- 主要CTAから問い合わせ完了まで計測できる。
- 文言版と問い合わせ完了を比較できる。
- 個人情報を分析イベントへ送っていない。
- 公開画像がWebPで、破損・参照切れがない。
- 本番計測スクリプトが200を返し、ランタイムエラーがない。
- 変更点、計測場所、次に数字を見る時期をユーザーへ伝えている。
